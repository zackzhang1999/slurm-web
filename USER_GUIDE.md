## 一、系统介绍

### 1.1 什么是 Slurm 集群监控系统？

Slurm 集群监控系统（Slurm Monitor）是一个专为 **Slurm 工作负载管理器** 设计的现代化 Web 监控平台。它提供了集群资源、作业状态、用户统计等多维度的实时监控和管理功能，帮助系统管理员和用户高效管理高性能计算集群。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| 🔴 **实时监控** | 基于 Socket.IO 的 WebSocket 实时数据推送，5秒自动刷新 |
| 📊 **资源统计** | CPU/GPU 核时使用统计，支持按用户、分区、时间段分析 |
| 🖥️ **Web Shell** | 基于浏览器的终端访问，支持系统 PAM 认证 |
| 📱 **响应式设计** | 适配桌面和移动设备，现代化的 Tailwind CSS 界面 |
| 🔐 **安全控制** | 管理员密码保护敏感操作 |
| 📝 **历史记录** | 支持历史作业查询和统计分析 |

## 二、功能模块

### 2.1 仪表盘 (Dashboard)

仪表盘提供集群整体运行状况的一览视图，包含以下核心指标：

#### 节点状态卡片
- **总节点数**: 集群中所有节点的数量
- **空闲节点**: 可接受新作业的节点（绿色）
- **已分配节点**: 正在运行作业的节点（黄色）
- **故障节点**: 不可用或离线的节点（红色）
- **混合状态节点**: 部分资源被占用的节点（紫色）

#### 作业状态卡片
- **总作业数**: 当前队列中的所有作业
- **运行中作业**: 正在执行的作业
- **等待中作业**: 队列中等待资源的作业
- **暂停作业**: 被管理员暂停的作业

#### 资源使用率
- **CPU 资源**: 总核心数、已分配核心、空闲核心及使用率进度条
- **GPU 资源**: 总 GPU 数、已分配 GPU、可用 GPU 及利用率进度条

#### 可视化图表
- **节点状态分布**: 饼图展示各类状态节点的比例
- **作业状态分布**: 饼图展示运行中、等待中、暂停作业的比例
- **CPU 资源使用**: 饼图展示已分配与空闲 CPU 的占比

#### 作业列表
- **运行中的作业**: 显示当前正在执行的作业列表
- **等待中的作业**: 显示队列中等待的作业列表

### 2.2 节点监控 (Nodes)

#### 节点详细信息表

| 字段 | 说明 | 示例值 |
|------|------|--------|
| 节点名 | Slurm 节点标识 | node01, gpu-node02 |
| 状态 | 当前状态 | idle, alloc, mix, down, drain |
| 分区 | 所属分区 | cpu, gpu, debug |
| CPU配置 | 分配/空闲/其他/总计 | 16/16/0/32 |
| 内存 | 总内存容量 | 128G |
| 空闲内存 | 当前可用内存 | 64G |
| 负载 | 系统负载 | 15.3 |
| GPU | GPU 资源 | gpu:a100:4 |
| 操作 | 管理操作 | 查看、Drain、Resume |

#### 节点状态说明

| 状态 | 含义 | 颜色标识 |
|------|------|----------|
| idle | 空闲，可接受新作业 | 🟢 绿色 |
| alloc | 已全部分配 | 🟡 黄色 |
| mix | 部分资源被占用 | 🟣 紫色 |
| down | 节点不可用 | 🔴 红色 |
| drain | 正在排空（维护中） | 🟠 橙色 |

#### 节点详情抽屉 ⭐ 新增

点击节点名称打开详情抽屉，展示以下信息：

**基本信息**
- 节点名、状态、所属分区、处理器架构

**资源配置**
- CPU 核心数、当前负载
- 总内存、空闲内存（自动转换为GB）
- CPU 配置详情（分配/空闲/其他/总计）
- GPU 资源（如有）

**Slurm 配置**
- 权重 (Weight)
- Slurm 版本
- 特征 (Features)

**操作按钮**
- **Drain 节点**: 将节点标记为维护状态（仅正常状态节点显示）
- **恢复节点**: 将节点从维护状态恢复（仅 Drain 状态节点显示）

> 💡 **提示**: 抽屉支持 ESC 键关闭，点击遮罩层也可关闭

#### 资源预留管理 ⭐ 新增

在节点监控页面下方可以管理资源预留。

**预留列表**
| 字段 | 说明 |
|------|------|
| 预留名称 | 预留的唯一标识 |
| 状态 | 预留是否激活 |
| 节点 | 预留的节点列表 |
| 开始时间 | 预留开始时间 |
| 结束时间 | 预留结束时间 |
| 持续时间 | 预留持续时长 |

**创建预留**
1. 点击"创建预留"按钮
2. 填写表单：
   - 预留名称（必填）
   - 节点列表（必填，支持 node001 或 node[001-010] 格式）
   - 开始时间（必填，支持 now 或具体时间）
   - 持续时间（必填，如 02:00:00）
   - 授权用户（可选）
   - 授权账户（可选）
3. 点击"创建"提交

**删除预留**
点击预留行中的 🗑️ 删除按钮，确认后删除预留。

#### 节点管理功能

**1. 排空节点 (Drain)**
将节点标记为维护状态，不再接受新作业，已运行的作业不受影响。
- 点击节点详情抽屉中的 "Drain 节点" 按钮
- 或点击节点列表中的操作按钮

**2. 恢复节点 (Resume)**
将节点从维护状态恢复为可用状态。
- 点击节点详情抽屉中的 "恢复节点" 按钮

> 🔐 **注意**: Drain/Resume 操作需要管理员密码验证

#### 作业详情抽屉 ⭐ 增强

点击作业 ID 打开详情抽屉，参考 `jobinfo` 命令展示详细作业信息：

**基本信息**
- 作业ID、状态、用户、账户、分区、QOS

**资源分配**
- 节点数、CPU 核心、预留内存

**时间信息**
- 提交时间、开始时间、结束时间
- 等待时间（Waited）
- 预留运行时间（Reserved walltime）
- 实际运行时间（Used walltime）

**CPU 使用详情** (已完成作业显示)
- CPU 总时间（Used CPU time）
- 用户态时间（% User/Computation）
- 系统态时间（% System/I/O）
- CPU 利用率

**内存使用详情** (已完成作业显示)
- 预留内存（Mem reserved）
- 最大内存使用（Max Mem used）
- 内存利用率

**磁盘 I/O** (已完成作业显示)
- 最大磁盘写入（Max Disk Write）
- 最大磁盘读取（Max Disk Read）

**完成状态** (已完成作业显示)
- 退出码（ExitCode）

**等待原因** (待运行作业显示)
- 调度原因（Reason）

**工作目录**
- 完整工作路径

**提交命令** (如 Slurm 版本支持)
- 完整的提交命令行

**操作按钮**
- **取消作业**: 取消正在运行或等待中的作业（需密码验证）
- **查看输出**: 显示作业输出文件路径

> 💡 **提示**: 
> - 抽屉支持 ESC 键关闭，点击遮罩层也可关闭
> - 不同状态的作业显示不同的信息区域
> - 运行中作业显示实时资源使用（通过 sstat）
> - 已完成作业显示历史统计（通过 sacct）

#### 历史任务 ⭐ 新增

作业管理页面新增"历史任务"标签页，可查看过去 30 天内的所有作业。

**功能特点：**

**1. 标签切换**
- **活跃作业**: 显示当前运行中、等待中、暂停的作业
- **历史任务**: 显示过去 30 天内已完成的作业

**2. 表格展示**
展示以下字段：
- 作业ID、名称、用户、账户、分区
- 状态（已完成/失败/已取消/超时）
- 退出码
- 节点数、CPU 核心、GPU 数量
- **预留内存** (Mem reserved)
- **最大内存使用** (Max Mem used)  
- **磁盘写入** (Max Disk Write)
- **磁盘读取** (Max Disk Read)
- 运行时间、提交时间

**3. 筛选和搜索**
- **状态筛选**: 所有状态 / 已完成 / 失败 / 已取消 / 超时
- **分区筛选**: 选择特定分区
- **用户搜索**: 输入用户名搜索

**4. 分页控制**
- 默认每页 50 条，可选 50/100 条每页
- 支持首页/上一页/下一页/末页导航
- 显示总任务数和当前范围

**5. 查看详情**
点击作业ID打开详情抽屉，展示完整的作业信息（与活跃作业共用详情抽屉）。

**数据来源：**
- 使用 `sacct` 命令获取历史作业数据
- 合并主作业和 `.batch` 步骤的资源使用数据
- 按提交时间倒序排列

> 💡 **提示**: 历史任务数据从 Slurm 数据库获取，可能需要几秒钟加载时间

#### 作业管理操作

**1. 批量选择**
勾选作业行前的复选框进行多选。

**2. 批量操作**
选中作业后，可执行以下操作：
- **批量取消**: 终止选中的作业
- **暂停作业**: 暂停运行中的作业（释放资源）
- **恢复作业**: 恢复已暂停的作业
- **挂起作业**: 挂起作业（保持资源）
- **释放作业**: 释放挂起的作业

输入管理员密码确认操作。

**3. 单个作业操作**
- 点击 🎬 暂停按钮：暂停运行中的作业
- 点击 ▶️ 恢复按钮：恢复已暂停的作业  
- 点击 ❌ 取消按钮：取消作业

**4. 搜索和筛选**
- 按状态筛选（全部/运行中/等待中/暂停）
- 按分区筛选
- 关键词搜索（作业名、用户名）

### 2.7 统计报表 (Statistics) ⭐ 核心功能

统计报表是系统的核心分析功能，提供多维度的集群使用分析。

#### 时间范围选择

点击页面右上角的时间选择器，支持：
- 📅 过去 24 小时
- 📅 过去 7 天（默认）
- 📅 过去 30 天
- 📅 过去 90 天

#### 2.7.1 集群使用率概览

四个关键指标卡片：
- ✅ **已完成作业**: 成功完成的作业数量
- ❌ **失败作业**: 执行失败的作业数量
- 📊 **成功率**: 完成作业占总作业的比例
- ⏱️ **总 CPU 时间**: 所有作业消耗的 CPU 时间（小时）

#### 2.7.2 用户资源使用情况 ⭐ 核心功能

展示每个用户的资源消耗详情：

| 列名 | 说明 | 示例 |
|------|------|------|
| 用户 | 用户名 | user01 |
| 作业数 | 该用户提交的作业总数 | 25 |
| CPU所用核时 | CPU 核心使用时间（分钟） | 5234.5 |
| 占比 | 该用户 CPU 使用占全集群的百分比 | 34.4% |
| 调用GPU的作业数 | 使用 GPU 资源的作业数量 | 5 |
| GPU所用核时 | GPU 核心使用时间（分钟） | 1234.5 |
| 占比 | 该用户 GPU 使用占全集群的百分比 | 35.7% |

**计算公式说明：**
- **CPU核时** = CPU核心数 × 运行时间(分钟)
  - 例如：使用 4 核运行 30 分钟 = 120 核时
- **GPU核时** = GPU卡数 × 运行时间(分钟)
  - 例如：使用 2 张 GPU 运行 60 分钟 = 120 核时

#### 2.7.3 作业分布图表

- **作业状态分布**: 饼图展示完成、失败、取消等状态比例
- **作业时长分布**: 柱状图展示作业运行时长的分布

#### 2.7.4 趋势分析

- **作业提交趋势(小时)**: 折线图展示每小时的作业提交数量
- **作业提交趋势(按天)**: 柱状图展示每日的作业提交数量

**智能显示逻辑：**
- 选择 24 小时：显示小时级趋势（X轴为 00:00-23:00）
- 选择 7天/30天/90天：显示日级趋势（X轴为 MM-DD）

#### 2.7.5 等待时间分析

显示作业在队列中的等待时间统计：
- 平均等待时间
- 中位数等待时间
- 最短/最长等待时间
- 等待时间分布饼图

#### 2.7.6 分区使用分析

分区使用详情表：

| 字段 | 说明 |
|------|------|
| 分区名 | 分区标识 |
| 总作业数 | 该分区的作业总数 |
| 已完成 | 成功完成的作业数 |
| 失败 | 执行失败的作业数 |
| 成功率 | 完成率百分比（带可视化进度条）|

#### 2.7.7 节点效率排行

节点资源利用效率排行表：

| 字段 | 说明 |
|------|------|

#### 2.7.8 用户排行

**按 CPU 时间排行**
- 显示 CPU 使用时间最长的用户
- 帮助识别资源消耗大户

**按作业数排行**

### B. 作业优先级详解 (AGE & Fairshare)

#### B.1 AGE 为什么显示为 0？

**现象**: 使用 `sprio -l` 查看时，AGE 列显示为 0。

**原因**: AGE 值太小被截断显示。

**计算公式**:
```
AGE = 等待时间 / PriorityMaxAge

例如:
- 作业提交 10 分钟 = 600 秒
- PriorityMaxAge = 14 天 = 1,209,600 秒
- AGE = 600 / 1,209,600 ≈ 0.000496 (显示为 0)
```

**查看原始值**:
```bash
# 显示更精确的 AGE 值
sprio -o "%i|%Q|%a"  # 格式: JOBID|PRIORITY|AGE原始值

# 示例输出
163|66|0.0000521  # AGE原始值很小但非零
```

**系统配置** (正常):
```
PriorityType=priority/multifactor  # 多因子优先级
PriorityWeightAge=1000            # AGE 权重
PriorityMaxAge=14-00:00:00       # 最大年龄 14 天
```

> ✅ **结论**: AGE 为 0 是正常现象，不影响作业调度。

#### B.2 Fairshare 值计算原理

Fairshare 是 Slurm **公平共享调度算法**的核心，防止某些用户垄断资源。

**计算目标**:
- 追踪每个用户的历史资源使用量
- 与应得份额比较，调整优先级

**计算公式**:
```
Fairshare = 2^(-U/S)

U = 用户实际使用的 CPU 时间 (加权历史)
S = 用户应得的份额 (由管理员配置)
```

**值含义**:

| Fairshare | 含义 | 优先级 |
|-----------|------|--------|
| 100 | 完全未使用资源 | 最高 |
| 67 | 使用略低于平均 | 中高 |
| 50 | 使用等于应得份额 | 中等 |
| 25 | 使用远超应得份额 | 较低 |
| 0 | 资源使用严重超标 | 最低 |

**加权历史计算** (指数衰减):
```
U = Σ(usage_i × decay_factor_i)

- 最近 7 天: 权重 50%
- 7-14 天前: 权重 25%
- 14-21 天前: 权重 12.5%
...
```

**查看 Fairshare**:
```bash
# 查看所有用户的 fairshare
sshare -l

# 查看特定用户
sshare -u username -l
```

**示例解释**:
```
用户 zack 的 Fairshare = 67

表示:
- 使用了一些资源，但未超过公平份额
- 获得中等偏上的调度优先级
- 主要优先级贡献来自 Fairshare (67)
```

**调整 Fairshare**:
```bash
# 修改用户份额权重
sacctmgr modify user zack set shares=200

# 修改 Fairshare 权重 (slurm.conf)
PriorityWeightFairShare = 1000  # 默认 100

# 修改半衰期 (slurm.conf)
PriorityDecayHalfLife = 7-00:00:00  # 默认 7 天
```
