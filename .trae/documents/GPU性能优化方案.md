# GPU信息获取性能优化方案

## 问题分析

当前通过 SSH 串行获取 GPU 信息的问题：

1. **串行执行**: 逐个 SSH 到每个 GPU 节点，每个节点完成后再执行下一个
2. **无缓存机制**: 每次页面刷新都重新 SSH 获取数据
3. **无超时优化**: SSH 连接慢的节点会阻塞整体响应

## 优化方案

### 方案：后台线程定期采集 + 缓存机制

#### 核心思路
- 使用后台守护线程定期采集 GPU 信息，存储在内存缓存中
- API 端点直接从缓存读取数据，响应时间从 O(n) 降为 O(1)
- 采集线程使用并发方式同时向多个节点请求

#### 实现步骤

1. **创建 GPU 信息缓存类** (`GPUCache`)
   - 使用 `threading.Lock` 保证线程安全
   - 存储 GPU 列表和采集时间戳
   - 提供 `get()` 和 `refresh()` 方法

2. **实现并发采集** (`collect_gpu_info_concurrent`)
   - 使用 `concurrent.futures.ThreadPoolExecutor` 并发 SSH 到多个节点
   - 设置超时时间，避免单节点阻塞
   - 支持部分失败（部分节点失败不影响整体）

3. **后台采集线程** (`GPUCacheUpdater`)
   - 继承 `threading.Thread`，守护线程
   - 定期（默认 5-10 秒）刷新缓存
   - 启动时立即采集一次

4. **修改 API 端点**
   - `/api/gpus`: 直接从缓存读取
   - `/api/gpu-processes`: 直接从缓存读取

5. **可选增强**
   - 支持手动刷新 API（如 `/api/gpus/refresh`）
   - 缓存过期时间戳返回给前端用于展示

#### 性能对比

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| API 响应时间 | O(n×SSH延迟) | O(1) 几乎瞬间 |
| 节点数增加影响 | 线性增长 | 无影响 |
| 页面刷新体验 | 随节点数变慢 | 始终流畅 |

## 代码修改清单

1. `app.py` - 添加 GPU 缓存相关代码
   - 添加 `GPUCache` 类
   - 添加 `GPUCacheUpdater` 后台线程
   - 修改 `api_gpus()` 和 `api_gpu_processes()` 使用缓存

## 风险控制

- 后台线程异常不影响主服务
- 缓存为空时同步获取一次
- 定期刷新保证数据新鲜度
